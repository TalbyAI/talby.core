## Introduction

This specification defines the architecture boundary for introducing C# source generators into Talby.Core. The goal is to keep shared contracts in `Talby.Core.Types` while placing generator behavior in a separate, reusable project that can be packaged and consumed by other solutions.

The first implementation slice is intentionally infrastructure-only. It must establish the solution shape, project boundaries, and publishable package structure before any production generated code is introduced.

## 1. Purpose & Scope

This specification covers the project, package, and solution boundaries required to support a reusable incremental source generator for Talby.Core.

In scope:

- The separation between shared contracts and generator implementation.
- The introduction of a dedicated generator project beside `Talby.Core.Types`.
- The use of an incremental generator based on `IIncrementalGenerator`.
- The packaging shape required for external reuse and publication.
- The minimum infrastructure required before generated output is introduced.

Out of scope:

- The concrete shape of emitted source for any domain feature.
- Consumer-specific APIs generated by the future generator.
- Runtime behavior changes in `Talby.Core.Types` beyond contract definition.
- Any packaging or deployment workflow unrelated to the generator package itself.

## 2. Definitions

| Term                      | Definition                                                                                                                                               |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Talby.Core.Types`        | The shared contracts project. It contains reusable type definitions and must not contain generator implementation.                                       |
| Generator project         | A separate project that contains the source generator implementation and is packaged for reuse.                                                          |
| Incremental generator     | A Roslyn generator that implements `IIncrementalGenerator` and processes inputs incrementally instead of using the classic full-rewrite generator model. |
| Analyzer package          | A NuGet package that delivers compiler-time behavior such as a source generator without becoming a runtime dependency of the consuming project.          |
| Infrastructure-only slice | The first implementation slice, limited to project wiring, package shape, and build validation. It does not require production emitted source.           |
| Publishable package       | A package that can be restored, referenced, and shipped to other projects outside this repository.                                                       |

## 3. Requirements, Constraints & Guidelines

- **REQ-001**: `Talby.Core.Types` shall remain a shared contracts project only.
- **REQ-002**: The generator implementation shall live in a separate project from `Talby.Core.Types`.
- **REQ-003**: The generator project shall implement `IIncrementalGenerator`.
- **REQ-004**: The generator package shall be reusable by other solutions, not limited to this repository.
- **REQ-005**: The first implementation slice shall establish generator infrastructure before any production emitted source is added.
- **REQ-006**: The solution shall include both `Talby.Core.Types` and the generator project once the boundary is established.
- **REQ-007**: The repository baseline shall remain `net10.0` with nullable reference types and implicit usings enabled unless a later spec explicitly changes that baseline.
- **REQ-008**: The contracts project shall not reference the generator project.
- **REQ-009**: The generator project may reference `Talby.Core.Types` if its implementation needs shared contracts.
- **REQ-010**: The generator package shall be packaged as an analyzer/source-generator package shape suitable for downstream compilation-time consumption.

- **CON-001**: The initial slice shall not require domain-specific generated APIs.
- **CON-002**: The initial slice shall not introduce a reverse dependency from contracts back to generator implementation.
- **CON-003**: The generator package shall remain publishable outside the repository boundary.
- **GUD-001**: Keep the first slice minimal until the packaging and build shape are verified.
- **PAT-001**: Use one-way dependency flow: generator depends on contracts, never the reverse.

## 4. Interfaces & Data Contracts

### Project Boundary

| Project            | Responsibility                                                   | Allowed Dependencies                                                | Disallowed Dependencies                                  |
| ------------------ | ---------------------------------------------------------------- | ------------------------------------------------------------------- | -------------------------------------------------------- |
| `Talby.Core.Types` | Shared contracts and reusable type definitions                   | Standard library, project-local contract dependencies               | Generator implementation projects                        |
| Generator project  | Incremental source generator implementation and package metadata | `Talby.Core.Types`, Roslyn generator APIs, packaging infrastructure | Any dependency on generated output as a runtime contract |

### Packaging Boundary

| Boundary            | Contract                                                                                                                                                         |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Consumer reference  | A downstream project must be able to consume the generator through package restore and compilation-time wiring, not by referencing generator internals directly. |
| Contracts reference | Consumers may reference `Talby.Core.Types` directly for shared types.                                                                                            |
| Generator reference | The generator package must be distributed in a form that compiler tooling can load as a source generator/analyzer package.                                       |

### Solution Boundary

The solution must explicitly include both the contracts project and the generator project so the repository can validate their relationship through a single build.

## 5. Acceptance Criteria

- **AC-001**: Given the repository solution is opened, when the solution is built, then `Talby.Core.Types` and the generator project both resolve correctly.
- **AC-002**: Given a consumer project references the generator package, when it is compiled, then the generator is available as a compile-time extension and not as a runtime-only library.
- **AC-003**: Given the first implementation slice is complete, when the generator project is inspected, then its architecture is incremental and based on `IIncrementalGenerator`.
- **AC-004**: Given the contracts project is built, when dependency analysis is performed, then it has no reference to the generator project.
- **AC-005**: Given the packaging shape is validated, when the generator package is produced, then it is publishable as a reusable package outside the repository.
- **AC-006**: Given the first slice is complete, when generated outputs are inspected, then no production feature-specific emitted source is required yet.

## 6. Test Automation Strategy

The testing strategy for the first slice focuses on build and wiring validation rather than generated feature output.

- **Test Levels**: Unit tests for generator wiring, build validation for project references, and package validation for publishability.
- **Frameworks**: Keep the repository's .NET baseline. If generator tests are added, place them in a dedicated test project rather than inside an existing production project.
- **Test Data Management**: Use minimal in-memory compilation inputs for generator smoke tests.
- **CI/CD Integration**: Validate the solution build and package shape in CI before adding feature-specific generator tests.
- **Coverage Requirements**: Coverage is not the primary quality gate for the first slice. Build and reference correctness are the primary gate.
- **Performance Testing**: Defer performance measurement until there is actual generated output to measure.

The first test slice should prove that the generator project loads, the contracts project remains isolated, and the package shape is consumable.

## 7. Rationale & Context

The design separates concerns that would otherwise become tangled quickly.

`Talby.Core.Types` should remain a contract surface because contracts are the stable integration point for consumers and for generated code. The generator implementation itself is a different kind of artifact: it must be packable, reusable, and loadable by compiler tooling. Combining those responsibilities into one project would make reuse harder and would blur the dependency direction.

An incremental generator is the right baseline because it is the modern Roslyn model for scalable source generation and keeps future feature slices compatible with incremental pipeline design.

The infrastructure-only first slice reduces risk. Packaging, project references, and solution inclusion are the parts most likely to fail if they are deferred until after emitted code exists. Validating those boundaries first makes later feature work safer.

## 8. Dependencies & External Integrations

### External Systems

- **EXT-001**: Roslyn compiler pipeline - required to host the source generator at compile time.

### Third-Party Services

- **SVC-001**: NuGet packaging - required so the generator can be published and consumed externally.

### Infrastructure Dependencies

- **INF-001**: .NET SDK 10 - required for the repository baseline.
- **INF-002**: Solution-level project inclusion - required to validate the contracts and generator projects together.

### Technology Platform Dependencies

- **PLT-001**: `net10.0` target framework - required to preserve the repository baseline.
- **PLT-002**: Nullable reference types - required to preserve current safety expectations.
- **PLT-003**: Implicit usings - required to preserve current project style.

## 9. Examples & Edge Cases

### Intended Project Relationship

```text
Consumer Project
  -> references Talby.Core.Types for shared contracts
  -> references the generator package as a compile-time analyzer/source generator

Talby.Core.Types
  -> shared contracts only
  -> no generator implementation dependency

Generator Project
  -> implements IIncrementalGenerator
  -> may reference Talby.Core.Types
  -> packages as a reusable analyzer/source-generator package
```

### Edge Cases

- If a consumer references only `Talby.Core.Types`, it should receive contracts but no generated behavior.
- If a consumer references only the generator package, it should still fail where shared contracts are required, because the contracts are a separate dependency boundary.
- If the generator project emits no production source in the first slice, that is acceptable as long as the infrastructure, packaging, and build shape are validated.
- If the contracts project starts referencing generator implementation, the architecture has been violated and the dependency direction must be corrected.

## 10. Validation Criteria

The specification is satisfied when all of the following are true:

- The solution includes the contracts project and the generator project.
- `Talby.Core.Types` remains isolated as shared contracts.
- The generator project is separate and incremental.
- The package shape is reusable and publishable.
- The first slice validates infrastructure and packaging before emitted source is introduced.
- No dependency cycle exists between contracts and generator implementation.
- The repository baseline remains `net10.0`, nullable enabled, and implicit usings enabled.
